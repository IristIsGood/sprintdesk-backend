问：你怎么实现用户认证？

"我用 Spring Security 加 JWT。用户注册时密码用 BCrypt 加密存进 PostgreSQL，登录时验证密码，通过后生成 JWT token 返回给前端。之后每个请求前端带着这个 token，后端验证有效才放行。整个流程是 stateless 的，不需要服务器存 session。"

问：为什么用 JWT 而不是 session？

"JWT 是 stateless 的，token 本身包含用户信息，服务器不需要存状态。适合 REST API 和未来扩展成微服务的场景。Session 需要服务器存状态，横向扩展时多台服务器要共享 session，复杂度更高。"

问：BCrypt 是什么？

"BCrypt 是一种单向哈希算法，专门用来存密码。它内置 salt，每次加密同一个密码结果都不一样，防止彩虹表攻击。验证时用 passwordEncoder.matches() 比对，不需要解密。"

1. How do you implement User Authentication?
Answer:

"I use Spring Security combined with JWT. When a user registers, I encrypt the password using BCrypt and store it in PostgreSQL. For login, I verify the credentials; if they match, the backend generates a JWT token and returns it to the frontend. After that, the frontend includes this token in the header of every request. The backend validates the token before allowing access. This whole process is stateless, meaning the server doesn't need to store any session data."

2. Why use JWT instead of Session?
Answer:

"JWT is stateless. Since the token itself contains the user information, the server doesn't need to keep track of who is logged in. This is perfect for REST APIs and makes it much easier to scale into microservices later. With Sessions, the server has to store state. If you scale horizontally (adding more servers), you have to share sessions between them, which adds a lot of complexity."

3. What is BCrypt?
Answer:

"BCrypt is a one-way hashing algorithm designed specifically for passwords. It has a built-in salt, which means even if two users have the same password, their hashes will look totally different. This prevents rainbow table attacks. When checking a password, I just use passwordEncoder.matches(). Since it's a one-way hash, you don't actually 'decrypt' it; you just compare the hashes."

"How does the Spring Security Filter Chain work?"
Answer:

"Every HTTP request goes through a Filter Chain. I implemented a custom filter by extending OncePerRequestFilter. This filter intercepts the request, extracts the JWT token from the Authorization header, and validates it.

Once validated, I store the user details into the SecurityContextHolder. Later in the chain, the AuthorizationFilter checks the SecurityContext. If it finds the authentication info, the request is allowed; otherwise, it returns a 403 Forbidden."

## N+1 问题文字总结

**什么是 N+1：**

你有 3 个项目，查项目列表时：
- 第 1 次 SQL：查所有项目 `SELECT * FROM projects`
- 第 2 次 SQL：查项目 1 的 owner `SELECT * FROM users WHERE id = 1`
- 第 3 次 SQL：查项目 2 的 owner `SELECT * FROM users WHERE id = 1`
- 第 4 次 SQL：查项目 3 的 owner `SELECT * FROM users WHERE id = 1`

3 个项目 = 4 次查询。100 个项目 = 101 次查询。这就是 N+1。

**为什么会发生：**

JPA 默认懒加载（`FetchType.LAZY`），只有真正访问 `project.getOwner()` 的时候才去查数据库。`toResponse()` 里调用了 `project.getOwner().getEmail()`，所以每个项目都触发一次。

**怎么修：**

用 `JOIN FETCH` 一次查询把 owner 一起拿回来：

```java
@Query("SELECT p FROM Project p JOIN FETCH p.owner WHERE p.owner.id = :ownerId")
List<Project> findByOwnerIdWithOwner(@Param("ownerId") Long ownerId);
```

一条 SQL 搞定，不管多少个项目都只查一次。

These are "must-know" topics for any backend interview. Here is how to express these concepts in professional, natural English that matches your technical level.

### 1. The Three Common Cache Issues

**Cache Penetration (缓存穿透)**
* **Definition:** Querying a key that doesn't exist in the database, causing every request to hit the DB directly.
* **Solution:** "To solve this, I **cache null values** with a short expiration time. Another advanced way is using a **Bloom Filter** to block invalid requests before they reach the database."

**Cache Breakdown (缓存击穿)**
* **Definition:** A "hot key" (very popular) expires, and a huge wave of requests hits the DB at the exact same moment.
* **Solution:** "I usually set hot keys to **never expire**, or I use a **mutex lock**. This ensures only one request hits the DB to update the cache while others wait."

**Cache Avalanche (缓存雪崩)**
* **Definition:** Many keys expire at the same time, crashing the database under the sudden load.
* **Solution:** "The best fix is to **add a random offset** to the expiration times. Instead of all keys expiring at 10 minutes, I set them to expire at **10 minutes plus a random 1 to 5 minutes**."



---

### 2. When to use Cache (and when not to)

**Good for Caching:**
* **Read-heavy data:** "Data that is read often but rarely changed, like **Project Details** or **User Profiles**."
* **High computation cost:** "Complex query results that take a long time to calculate."
* **Eventual consistency:** "When it's okay if the data is a few seconds out of date."

**Bad for Caching:**
* **Write-heavy data:** "Data that changes constantly, like **Action Logs**."
* **Strong consistency required:** "Sensitive data like **Bank Balances** or **Real-time Inventory**."
* **Small datasets:** "If the table is tiny, querying the DB is already fast enough."

### 3. MongoRepository vs. JpaRepository

**The Comparison:**
The usage is almost identical because of **Spring Data**, but there are key differences in implementation.

* **JPA Repository:** `public interface ProjectRepo extends JpaRepository<Project, Long> {}`
* **Mongo Repository:** `public interface LogRepo extends MongoRepository<ActivityLog, String> {}`

**Answer:**
